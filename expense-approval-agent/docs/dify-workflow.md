# Dify工作流设计

## 节点顺序

1. 开始节点：接收报销文本或表单字段。
2. LLM结构化提取：输出员工编号、费用类型、金额、发票号码、日期和用途。
3. 参数校验分支：缺失字段直接生成补充材料清单。
4. 知识检索：检索对应费用制度，仅用于解释依据。
5. HTTP请求：POST到 `/api/precheck`，由Python规则层查询SQLite并生成审批建议。
6. 条件分支：按 `decision` 和 `need_human_review` 分发。
7. 主管审批：1000至5000元或指定业务风险。
8. 财务复核：超过5000元、预算不足、重复发票或员工异常。
9. 结果节点：返回审批结论、原因、缺失字段、下一步和制度来源。

## 关键边界

- Dify的LLM节点不能自行修改金额、员工身份或预算结果。
- HTTP节点返回失败时不得把申请描述成已通过。
- 所有人工审批节点必须保留，不以Prompt绕过。
- API密钥放在Dify凭证或环境变量中，不写进工作流正文。

## 面试讲解重点

与客服Agent不同，报销审批不能主要依赖模型自由判断。这里采用“LLM负责非结构化理解，规则引擎负责确定性决策，数据库负责事实，人工负责高风险审批”的组合。
