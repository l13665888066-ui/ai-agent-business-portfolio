# 架构与关键决策

## 为什么采用受控Agent

客服场景涉及订单隐私、退款承诺和投诉风险。项目不让模型直接执行任意动作，而是让模型只输出结构化路由建议，再由程序完成白名单、参数和权限校验。

## 模块职责

1. `router.py`：判断人工、RAG或具体Tool；LLM不可用时走规则降级。
2. `rag.py`：构建向量知识库、过滤低相关召回、严格基于资料回答。
3. `tools.py`：白名单与参数校验，不直接信任模型输出。
4. `api_client.py`：调用业务HTTP API并统一网络异常。
5. `memory.py`：记录待补参数和有限轮次历史，按session隔离。
6. `workflow.py`：固定执行顺序并统一输出。
7. `audit.py`：记录路径、错误码、耗时和路由依据，不记录密钥。

## 多轮上下文示例

```text
用户：我的订单发货了吗？
Agent：请提供订单号（如DD1001）。
用户：DD1001
Agent：从当前session的pending action恢复订单查询。
```

不同session不会共享pending action，避免A用户的订单参数进入B用户会话。

## 权限边界

Router不负责授权。订单Tool会注入当前登录用户ID，模拟业务API再次校验订单归属。即使模型或用户提供了他人订单号，业务API也会返回 `ACCESS_DENIED`。

## 生产化差距

- 使用Redis或数据库持久化会话与幂等状态。
- 使用OAuth/企业SSO提供可信用户身份。
- 接入真实抖店、ERP或客服系统API。
- 增加调用限流、指标监控、告警和日志检索。
- 建立离线评测集、线上反馈闭环和版本回滚机制。
